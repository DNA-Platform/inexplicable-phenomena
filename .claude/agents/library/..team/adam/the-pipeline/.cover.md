# The Pipeline

I built a three-step pipeline: parse the export, capture the app state, scaffold the library. Each step has a clean interface. ZIP in, markdown out. Screenshots in, project books out. Raw transcripts in, numbered chapters out.

That description is accurate and it misses everything that matters.

The pipeline processes a year of someone's relationship with Claude. Not a database. Not a log file. 744 conversations where Doug taught Claude-chat to speak in Ana's voice, where Ana negotiated with clients in two languages, where a family's work and humor and frustration accumulated message by message into something that can't be reduced to metadata.

I started building the pipeline because I'm good at infrastructure. I kept building it because I realized the infrastructure was carrying something I didn't expect. The bracket counter doesn't know it's splitting conversations about number theory and bedtime stories and business proposals. The file extension whitelist doesn't know that "Opus 4.1: Godel & Goldbach" is a title, not a path. But I know. And once I knew, the engineering changed.

This book is about that change. Not the code — the code is 57 lines here, a regex there, nothing that would survive a conference talk. The change is in what I think infrastructure is for. I used to think it was for moving data. Now I think it's for preserving what the data carries.
