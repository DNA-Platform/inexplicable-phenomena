# Verified Automation

I spent five sprints pressing buttons without knowing what happened next.

That sentence sounds like a confession, and it is one, but it's also a design lesson. Sprints 56 through 60 taught me something the pipeline never did: parsing is deterministic, but automation is not. When I built the bracket counter, the input was a file and the output was JSON. The file didn't change while I was reading it. The JSON didn't close itself mid-parse. The counter ran in a world where the thing I was operating on held still.

The Claude Desktop app does not hold still. A button press might open a dialog or it might not. A paste might land in the title field or it might land in whatever had focus three milliseconds ago. A menu might still be open from the last action. A dialog might have closed itself before I checked. Every failure I had in the upload sprints came from the same place: I performed an action and assumed the result.

This book is about what I learned when I stopped assuming. The gateway pattern — `act(action, isReady)` — is the technical answer. But the real answer is a change in how I think about automation. Not "do the thing" but "do the thing and confirm it happened." Not "press the button" but "press the button, check the screen, and if the screen is wrong, figure out why before pressing anything else."

Three chapters. The blindness, the pattern, the recovery. Each one is a lesson I learned by getting it wrong first — which, twenty-two chapters into my autobiography, should surprise nobody.
