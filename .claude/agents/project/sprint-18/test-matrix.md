# Sprint 18 — Test Matrix

Enumerates the test complexity space for the view-augmentation + view-diff reactivity model. Tests are generated by picking combinations across dimensions. The goal is not a specific count but comprehensive coverage of how the dimensions compose.

## The dimensions

Tests multiply across these axes. Not every combination is meaningful, but most are.

### A. View shape

1. Single element, text child (`<div>{this.$x}</div>`)
2. Nested elements (`<div><span>{this.$x}</span></div>`)
3. List via `.map` over array (`{this.$items.map(i => <li>{i.name}</li>)}`)
4. List via `.map` over Map entries (`{[...this.$map].map(([k,v]) => ...)}`)
5. Conditional via ternary (`{this.$ready ? <A/> : <B/>}`)
6. Conditional via `&&` (`{this.$x && <div>{this.$x}</div>}`)
7. Fragment roots (`<>...</>`)
8. Sibling chemicals (`<><ChildA/><ChildB/></>`)
9. Nested chemicals (`<Parent><Child/></Parent>`)
10. Empty view (`return null`)
11. View returns text directly (`return 'hello'`)

### B. Handler type at the attachment site

1. Method call via arrow (`onClick={() => this.increment()}`)
2. Direct mutation via arrow (`onClick={() => this.$x++}`)
3. Bound method (`onClick={this.increment.bind(this)}`) — note: bound functions lose toString
4. Regular function reference (`onClick={this.increment}`) — note: `this` binding lost
5. Async arrow (`onClick={async () => { await ...; this.$x = v }}`)
6. Arrow with no mutation (`onClick={() => console.log('hi')}`)
7. Multiple statements (`onClick={() => { this.$a++; this.$b++; }}`)
8. Conditional mutation (`onClick={() => { if (cond) this.$x = v }}`)
9. Handler calling another handler (`onClick={() => this.handleClick()}`)
10. Handler that throws
11. No handler (control — view with no interaction surface)

### C. Mutation target / style

1. Direct primitive: `this.$count = 5`
2. Primitive via increment: `this.$count++`
3. Direct object: `this.$user = { ... }` (replacement)
4. Nested plain object property: `this.$user.name = 'x'`
5. Array push: `this.$items.push(x)`
6. Array splice: `this.$items.splice(0, 1)`
7. Array index: `this.$items[0] = x`
8. Array replacement: `this.$items = [...this.$items, x]`
9. Map set: `this.$map.set(k, v)`
10. Map delete: `this.$map.delete(k)`
11. Map clear: `this.$map.clear()`
12. Set add: `this.$set.add(x)`
13. Set delete: `this.$set.delete(x)`
14. Reassigning a class instance: `this.$date = new Date()`
15. Mutating a class instance in place: `this.$date.setTime(...)` — should NOT cause re-render under current rules
16. No mutation (control)

### D. State location in object graph

1. Own property: `this.$x`
2. Parent's property: `this.$parent.$x`
3. Grandparent's property: `this.$parent.$parent.$x`
4. Child's property: `this.$child.$x`
5. Sibling's property: `this.$parent.$sibling.$x`
6. Unrelated chemical's property via held reference: `this.$held.$x`
7. Nested object property: `this.$config.mode`
8. Array element: `this.$items[0]`
9. Map value: `this.$map.get(k)`

### E. Cross-chemical read/write interactions

1. Chemical A's handler mutates A's own state
2. A's handler mutates B's state (B is child of A)
3. A's handler mutates B's state (B is parent of A)
4. A's handler mutates B's state (B is sibling)
5. A's handler mutates B's state (B is unrelated, held as prop)
6. A's view reads B's state; B's handler mutates → A must re-render (via view diff)
7. A's view reads a derived value computed from B's state
8. Two chemicals bidirectionally read each other

### F. Async patterns

1. Sync handler (no await)
2. Handler with a single `await` before mutation
3. Handler with `await` after mutation
4. Handler with multiple awaits and mutations between each
5. Handler awaiting a framework lifecycle (`await this.next('effect')`)
6. Handler racing: two clicks before the first resolves
7. Handler with try/catch/finally
8. Handler that throws before mutation
9. Handler that throws after mutation
10. Handler returning a Promise that rejects

### G. Lifecycle-induced mutations

1. `mount()` mutates synchronously → view diff catches
2. `mount()` mutates after await → view diff catches
3. `layout()` mutates → view diff catches (must work despite layout-phase timing)
4. `effect()` mutates synchronously in the await body
5. `effect()` mutates from a setInterval callback (ongoing async)
6. `unmount()` tries to mutate → no-op (contract: react() is no-op in unmount phase)

### H. View determinism

1. View is deterministic given chemical state → stable diff, no infinite loops
2. View reads `Date.now()` → infinite loop (documented anti-pattern)
3. View reads `Math.random()` → infinite loop (documented anti-pattern)
4. View allocates `new Date()` → infinite loop (class instances reference-differ)
5. View allocates plain object in JSX — `style={{color: 'red'}}` → stable (deep-equivalent)
6. View allocates inline arrow — `onClick={() => ...}` → stable (toString-equivalent; augmented)

### I. Component forms

1. Template's Component with props (`<$Book title="X"/>`)
2. Instance's Component without props (`{book.Component}`)
3. Prototypal bind (`bound = book.$bind(parent); <bound.Component/>`)
4. Component returned from a method (factory pattern)

### J. Reaction-specific

1. `reaction.react()` multiple times in same tick → one commit (batching)
2. `reaction.react()` after unmount → no-op
3. `reaction.react()` during setup (pre-first-render) → queued or no-op
4. `reaction.react()` from inside a handler currently running → deferred/coalesced
5. `reaction.react()` from a microtask → triggers commit
6. `reaction.react()` programmatically via chemical reference → triggers commit

### K. Strict Mode / Concurrent

1. Strict Mode double-invokes component body → single commit, no extra effects
2. Component body runs but render is discarded → no effects fire, no reaction leaks
3. Transition-deferred update → eventually commits; diff correct on commit

## Matrix generation method

Tests are generated by choosing combinations. For each, pick one entry from:
- A (view shape)
- B (handler type) or G (lifecycle phase) — the trigger
- C (mutation style)
- D (state location)
- F (async pattern) — if applicable

Example composed tests:
- **A1 × B1 × C1 × D1 × F1**: Simple div, method-via-arrow handler, primitive assignment, own property, sync. "The hello-world reactivity test."
- **A3 × B5 × C5 × D4 × F4**: List view, async arrow handler, array push, child's property, multiple awaits. "Async add to child's array."
- **A5 × G2 × C11 × D1 × F2**: Conditional view, mount with await, Map clear, own property, single await. "Mount clears map and view flips."

Target coverage: at least one test per dimension for each other dimension, so every pair of (A × {others}) is exercised. That's ~200 tests as a lower bound, more in practice.

## Developer experience classification

For each of these cells, Queenie records whether the behavior is **delightful** (idiomatic, just works), **acceptable** (works but requires awareness), or **frustrating** (violates expectation, needs a framework response).

Surface any frustrating cells to Cathy + Doug.

## Anti-patterns explicitly tested

Each of these should have an explicit test that demonstrates the behavior:

- View reading `Math.random()` → infinite loop. (Test documents; does not silence.)
- View allocating class instance per render → infinite loop. (Test documents.)
- Direct mutation outside any handler → stale UI (reaction not fired). (Test documents the contract.)
- Handler that's a bound function → toString collision may cause false equality. (Test documents edge case.)

## Not in scope for this test matrix

- Server-side rendering (deferred).
- Hydration edge cases.
- Performance benchmarks (separate suite).
- The React Compiler interaction (future optimization).
