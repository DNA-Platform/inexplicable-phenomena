# Review: Team system & project infrastructure

**Date:** 2026-03-31
**Reviewers:** Arthur (Architect), Libby (Librarian)
**Scope:** `.claude/team/` (roles, abilities, agents, registry), `.claude/project/` (tracker, sprints)

---

**Arthur:** My diagnostic question for this review: *What depends on this, and what does it depend on?*

Everything depends on this. Every skill, every sprint, every `/review`, every story execution loads from these directories. The team system is the context-loading infrastructure for the entire project. Nothing depends on it in the code-import sense — it's not `require()`'d — but it's the most load-bearing content in the repo because it shapes how Claude thinks about every task.

**Libby:** My diagnostic question: *Who will read this, and what do they need to do next?*

The readers are future Claude sessions — and Doug when he's reviewing what we've built. A session reads the project tracker to find where we are, loads agent/role/ability files to adopt a perspective, then executes. If these files are inconsistent, the session starts with a broken mental model. If they're clear, the session is productive from the first tool call.

## Understanding

**Arthur:** The system has four layers:

1. **Abilities** (18 files) — Domain knowledge documents. Specific techniques: API calls, regex patterns, timing constants, gotchas. Leaf nodes — they reference nothing in the team system.

2. **Roles** (6 files) — Perspectives on code. Each has: a diagnostic first question, anxieties, a mantra, abilities to load, source files to read, and a "How I become X" narrative. Roles reference abilities (downward only).

3. **Agents** (5 files + registry.json) — Named entities applying one role to specific paths. Agents reference roles (downward only). The registry is the lookup index.

4. **Project** (tracker + sprint directories) — Work organization. Sprint plans decompose into epics/stories/spikes. Kanban boards track state. The tracker is the entry point for session resumption.

The dependency direction is strict: **agents → roles → abilities**. No upward references. This was enforced in the organize pass — "Used by:" tags were removed from all 18 ability files.

The data flow at execution time: skill invocation → agent lookup (registry) → role load → ability load → source file read → task execution.

**Libby:** The project layer sits adjacent to the team layer, not above it. Sprint plans reference agents by name (who owns a story), but agents don't know about sprints. The project tracker references roles and agents in its documentation tables, but as navigational links — not as dependencies. This means the project system can evolve (new sprint formats, different board structures) without touching the team system.

## Strengths

**Arthur:** Here's what genuinely works, and why it works from what we know about LLMs:

- **Diagnostic first questions are the single most valuable element.** They force a frame before analysis begins. "What depends on this?" immediately anchors in dependency thinking. "What concept does this encode?" anchors in fidelity thinking. These questions are more valuable than the anxieties list because they set the *lens*, not just the checklist.

- **One role per agent is correct.** Compound roles degrade quality (Wang et al., NAACL 2024). When I load Framework Engineer, every choice gets evaluated against "is the abstraction faithful?" When I load Architect, every change gets evaluated against "does this dependency graph make sense?" These are genuinely different attention patterns that dilute when merged.

- **Board-as-truth handles context interruption.** LLM conversations get compacted, sessions end, context vanishes. The kanban board survives all of this because it's a file. The `/sprint resume` pattern — read the board, find what's in progress, continue — is the correct architecture for multi-session work.

- **Ability/role separation prevents cross-contamination.** Abilities encode *what you know*; roles encode *what you care about*. This separation means abilities can be shared across roles without blending perspectives.

**Libby:** From a documentation perspective:

- **The role files' "How I become X" sections are genuine documentation of a real phenomenon.** They describe what happens when specific knowledge is loaded into an LLM's context — how it changes attention, what it makes the model check. This isn't roleplay flavor text; it's operational documentation of context effects.

- **The project tracker as entry point is correct.** "Read this file first when resuming work" — that single instruction at the top of `index.md` is the most important line in the project system. Every other organizational pattern only works if sessions actually find the starting point.

- **Markdown reference links throughout create a navigable graph.** The `<!-- citations -->` convention at the bottom of every file means you can trace from any document to its dependencies. This is more useful than a central index because it works from wherever you're reading.

## What "results" does this system produce?

**Arthur:** Let me catalogue concretely:

1. **Consistent attention across sessions.** Without roles, each session approaches code from whatever angle seems obvious. With roles, the same anxieties get checked every time. This is reproducibility — the thing that makes engineering different from ad-hoc problem solving.

2. **Ownership clarity.** `/responsible` answers "who cares about this file?" instantly. This prevents the diffusion of responsibility that happens when an LLM treats all code as equally its concern. Focused ownership produces deeper analysis than broad coverage.

3. **Resumability.** A new session can reconstruct where we are — what sprint, what's done, what's blocked, what's next — from file reads alone. No human needs to brief the model.

4. **Structured review.** Reviews grounded in a specific perspective find things generic reviews miss. The sprint-2 reviews (by the now-deleted Tap/Sift/Pace) found timing issues, idempotency violations, and log rotation needs — specific bugs that a "review this code" instruction wouldn't have surfaced.

5. **Sprint discipline.** Decomposing work into stories with dependencies prevents the "do everything at once" failure mode that wastes tokens and produces incomplete results.

**Libby:** There's a sixth result that's easy to miss:

6. **Shared vocabulary.** The team system creates a lexicon that persists across sessions. When Doug says "have Cathy look at this," a future session knows exactly what that means: load framework-engineer, focus on library/chemistry/**, bring the type-fidelity lens. Without the team system, that sentence is meaningless to a new session.

## Structural changes (second pass)

### Ability compression — 18 files → 5 files

**Arthur:** The original review noted abilities as "maintenance burden." The deeper question Doug raised: does it need to be one file per ability? The answer is no. Abilities cluster by domain — the 6 relay transport abilities always load together, the 4 framework design abilities always load together. Grouping by co-occurrence reduces file count without losing content.

**Cathy:** From my perspective as framework + frontend engineer: the four abilities I load (oop-patterns, functional-programming, scheme-and-self, framework-architecture) are now one file. When I load `framework-design.md`, I get the full conceptual foundation in a single read. The sections are navigable by heading if I need to focus on one area.

### Multi-role agents restored

**Arthur:** The original review enforced single-role-per-agent based on Wang et al.'s compound persona research. Doug correctly identified this as a misapplication: the paper's concern is about conflicting *personas* (pirate-doctor), not additive *checklists*. Loading two structured lists of anxieties is a bigger checklist, not a confused identity.

More importantly: if agents only have one role, then agents ARE roles with paths attached. The agent layer adds nothing — it's just an indirection. Multiple roles per agent means the agent is the *intersection* of perspectives applied to specific code. That intersection IS the unique perspective the agent layer exists to provide.

**Cathy:** My code proves the need empirically. `particle.tsx` uses `Object.setPrototypeOf` (framework concern) and returns `ReactNode` (frontend concern) in the same file. I now load both framework-engineer and frontend-engineer roles, which means I check for both type fidelity AND component API clarity. The diagnostic questions complement: "What concept does this encode?" and "What does the developer using this write?" Both matter for every file I own.

## Summary

**Arthur:** Two structural changes define this review. First, ability compression (18 → 5 files) reduced the layer's footprint while fixing 6 orphaned abilities and preserving all content. The groupings follow co-occurrence patterns — abilities that load together live together. Second, multi-role agents restored the agent layer's purpose: agents are not roles with paths, they are the *intersection of perspectives* applied to specific code. The dependency tree is now: agents → roles → abilities (grouped by domain) → nothing. Clean, shallow, unidirectional.

**Cathy:** The changes match my reality. I work on code that is simultaneously framework design and frontend engineering. Having both lenses loaded means I catch both categories of issues in a single review pass. The grouped ability file means I load one file instead of four and get everything I need. The system now describes what was always true about the work.

**Libby:** The review file moved from sprint-4/reviews/ to project/reviews/ because this review isn't about sprint-4 — it's about the team system itself. Historical sprint reviews (tap.md, sift.md, pace.md in sprint-2) stay where they are. Project-level reviews live at the project level. The structure matches the content.
